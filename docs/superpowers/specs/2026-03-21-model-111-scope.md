# Modelo 111 V1: documento de decisión de alcance

Fecha: 2026-03-21  
Estado: decidido para la Task 5.3  
Tipo: gate de producto y de implementación

## 1. Decisión ejecutiva

La decisión para V1 es:

- soportar `Modelo 111` solo como `resumen trimestral manual`
- no soportar cálculo automático de `111` desde transacciones o `payroll_placeholder`
- no hacer `payroll import` parte del alcance V1

Consecuencia directa:

- el roadmap MVP sigue cerrando con `303`, `115`, periodos, cierre y advisor-ready exports
- `111` no se vende como automatizado
- `190` queda bloqueado hasta disponer de fuente de nómina fiable y trazable

## 2. Problema que resuelve esta decisión

El repo ya asume una S.L. española con empleados y alquiler con retención. También deja fijado que hoy no existe una fuente de nómina fiable dentro del producto.

Con ese contexto, prometer un `111` completo a partir del dato actual sería incoherente por tres motivos:

- el contrato fiscal congela nómina como `payroll_placeholder` bloqueado
- el dataset dorado excluye `111` y deja nómina fuera de `303` y `115`
- el roadmap MVP excluye explícitamente `111` automatizado sin fuente de nómina

La decisión necesaria no es si `111` existe o no, sino qué forma mínima puede tener sin crear una falsa sensación de cálculo automático.

## 3. Alcance V1 normalizado

En alcance para V1:

- un borrador trimestral de `Modelo 111` cargado manualmente por el usuario
- captura de un resumen agregado trimestral procedente de una fuente externa de nómina o asesoría
- estado explícito de `manual`, `pendiente de evidencia` o `listo para revisar`
- conservación de evidencia adjunta o referencia documental externa
- visibilidad clara de que el producto no ha calculado las retenciones de empleados

Fuera de alcance para V1:

- cálculo automático de retenciones de trabajo
- inferencia de `111` desde movimientos bancarios o transacciones genéricas
- uso de `payroll_placeholder` como fuente computable para `111`
- desglose por empleado dentro del producto
- importadores de nómina
- reconciliación automática nómina -> trimestre
- generación de `190`
- soporte generalista de todos los supuestos posibles de `111` fuera del contexto confirmado

## 4. Opciones evaluadas

### Opción A: payroll import first

Descripción:

- no existe `111` hasta integrar una fuente de nómina estructurada y fiable
- el producto solo desbloquea `111` cuando puede importar detalle suficiente por empleado o por asiento de nómina

Ventajas:

- máxima trazabilidad
- menor riesgo de declarar importes agregados sin soporte interno
- deja mejor preparado el camino hacia `190`

Riesgos normativos y operativos:

- bloquea la entrega de cualquier soporte `111` hasta resolver una dependencia que hoy no existe
- introduce riesgo de roadmap porque la integración de nómina no está aprobada ni acotada en este repo
- desplaza el problema a un conector o formato externo no definido

Veredicto:

- técnicamente sólido
- no es la opción más pragmática para cerrar el roadmap aprobado

### Opción B: resumen trimestral manual

Descripción:

- el producto permite registrar manualmente el total trimestral del `111` a partir de una fuente externa fiable
- el dato se trata como resumen declarado o revisado, no como cálculo del sistema

Ventajas:

- encaja con el estado real del producto
- no contradice el contrato fiscal ni el dataset dorado
- permite cubrir el hueco operativo del trimestre sin prometer automatización inexistente
- mantiene bajo el coste de implementación y de soporte

Riesgos normativos y operativos:

- riesgo de error humano en la transcripción
- no existe trazabilidad interna por empleado
- no sirve como base suficiente para `190`
- si la UI comunica mal el alcance, el usuario puede asumir que el sistema “ha calculado” el modelo

Veredicto:

- es la mejor opción para V1 si se etiqueta y limita de forma estricta

### Opción C: ambos desde V1 con límites

Descripción:

- V1 admite resumen manual y deja abierto un camino paralelo de importación de nómina

Ventajas:

- flexibilidad futura
- reduce riesgo de rediseño si el import llega pronto

Riesgos normativos y operativos:

- mezcla dos fuentes de verdad desde el primer corte
- complica validaciones, UX, soporte y auditoría
- difumina la promesa comercial
- aumenta el riesgo de que un trimestre combine datos manuales y parciales importados sin criterio fuerte

Veredicto:

- no recomendable como decisión V1
- puede ser un camino de evolución posterior, no una promesa inicial

## 5. Recomendación final

Se aprueba la `Opción B: resumen trimestral manual` como único alcance V1 para `Modelo 111`.

Razón de producto:

- resuelve la necesidad real de una S.L. con empleados sin fingir que TaxHacker ya entiende nómina
- mantiene el roadmap MVP centrado en lo ya aprobado y verificable
- evita convertir una carencia de fuente en una deuda de confianza con el usuario

Razón técnica:

- evita contaminar el dominio fiscal canónico con importes de nómina no soportados por facts fiables
- preserva la regla de `payroll_placeholder` como rastro bloqueado y no computable
- permite modelar `111` como una capa de resumen trimestral separada del core transaccional

## 6. Decisiones de diseño derivadas para ingeniería

- `111` V1 debe modelarse como `resumen trimestral manual`, no como agregado derivado de `fiscal_document_header` o `fiscal_tax_line`
- los `payroll_placeholder` deben seguir en `blocked` y no pueden alimentar automáticamente el borrador trimestral
- el quarter puede mostrar una alerta de “obligación 111 existente”, pero no puede mostrar importes calculados salvo que provengan de entrada manual explícita
- si en el futuro existe import de nómina, esa fuente deberá convivir como modo alternativo y mutuamente excluyente con el resumen manual para un mismo trimestre

## 7. Criterios de aceptación y fronteras de fallo

Aceptación mínima para considerar bien implementado este alcance:

- el producto etiqueta `111` como `manual` o equivalente visible para el usuario
- el usuario puede registrar un resumen trimestral sin que el sistema afirme haberlo calculado
- el flujo exige referencia o evidencia externa de soporte
- los `payroll_placeholder` no modifican importes del `111`
- el quarter queda claramente marcado como `111 no automatizado`
- `190` sigue bloqueado en producto y en planificación técnica

Frontera de fallo:

- cualquier pantalla o export que presente `111` como cálculo automático
- cualquier uso de nómina no estructurada como fuente computable
- cualquier implementación que intente derivar `190` desde resúmenes trimestrales manuales
- cualquier mezcla de importes manuales con importes inferidos del core transaccional sin fuente de nómina aprobada

## 8. Riesgos por opción y mitigación recomendada

Riesgos de la opción aprobada:

- error humano al transcribir
- evidencia externa inconsistente
- interpretación comercial equivocada del alcance

Mitigaciones obligatorias:

- copy explícito: `resumen manual`, `no calculado por TaxHacker`
- campo de evidencia o referencia documental obligatorio
- estado de revisión diferenciado de los modelos calculados como `303` y `115`
- bloqueo explícito de `190`

## 9. Criterio para desbloquear una implementación más automatizada

La automatización futura de `111` solo se desbloquea si existe una fuente de nómina que cumpla como mínimo:

- identificador estable por empleado o perceptor
- periodo de devengo o liquidación claramente definido
- importes sujetos a retención con semántica estable
- retención practicada con precisión de céntimo
- capacidad de reconciliar el total trimestral con el detalle origen
- contrato de importación repetible y testeable con fixture estable

La automatización no se considera desbloqueada si solo existe:

- PDF suelto
- extracto bancario
- OCR de recibos de salario
- importes agregados sin detalle trazable

## 10. Implicaciones para Modelo 190

La implicación es directa:

- `190` no entra en alcance después de este gate
- `190` sigue condicionado a una fuente de nómina estructurada y a detalle anual por perceptor
- un resumen manual trimestral de `111` no proporciona granularidad suficiente para `190`

Por tanto, el roadmap debe leerse así:

- `111` V1: soporte manual y claramente no automatizado
- `190`: deferred y bloqueado hasta resolver import de nómina o fuente equivalente con trazabilidad anual

## 11. Supuestos y dependencias explícitas

Supuestos confirmados por las fuentes:

- la sociedad objetivo tiene empleados
- el producto actual no tiene fuente de nómina fiable
- `303` y `115` sí tienen base de datos y contrato fiscal para avanzar antes

Dependencias para una fase posterior:

- definición de fuente externa de nómina aceptada
- fixture de importación de nómina
- decisión de producto sobre si el modo futuro será `manual o importado`, pero nunca mezcla silenciosa

## 12. Recomendación de handoff

Handoff recomendado para ingeniería:

- tratar `Modelo 111` V1 como un track documental y operativo separado del cálculo fiscal automático
- no reutilizar `payroll_placeholder` como si fuera fuente de retención válida
- mantener `190` fuera de cualquier plan de implementación hasta cerrar un contrato de importación de nómina

Con esta decisión, el roadmap queda cerrado de forma coherente: el producto puede ayudar en `111` sin vender un motor de nómina que hoy no existe.
